# Full Docusaurus documentation site for the coding agent

## Goal

Stand up a complete, runnable Docusaurus site under `website/` that documents the coding
agent described in `PLAN.md` — modelled on pi.dev's documentation IA — plus a `Taskfile.yml`
(go-task) so the docs can be installed, built, and served with one command.

## Scope & grounding

- Single source of truth for content: `PLAN.md` (the `src/*.py` files are still empty stubs).
- **Target: full pi.dev parity** (user decision, 2026-06-19). Every pi.dev sidebar topic is
  covered or explicitly marked N/A with a rationale in `differences-from-pi.md`.
- IA mirrors pi.dev's grouping: Getting Started → Architecture → Tools → Concepts →
  Customization → Operations → Programmatic Usage → Guides → Advanced → Reference →
  Platform Setup → Contributing, recast for this Python-from-scratch project.
- Docs use Docusaurus autogenerated sidebar: one `_category_.json` per folder +
  `sidebar_position` frontmatter per page. No hand-maintained `sidebars.js` entries.

## Order of operations

1. Scaffold the site (package.json, config, sidebar, css, landing page). **[done]**
2. Write `_category_.json` for every section. **[in progress]**
3. Write `intro.md` myself to fix tone + a shared STYLE GUIDE for authors.
4. Fan out section authoring to parallel subagents, each handed the relevant `PLAN.md`
   excerpts + the style guide + exact filenames/frontmatter to produce.
5. Add `Taskfile.yml` with docs tasks.
6. Verify: `task docs:build` produces `website/build/` with no broken-link errors.

## Style guide (passed to every author agent)

- Frontmatter on every page: `sidebar_position`, `title`, `description`.
- Audience: an engineer learning how a coding agent works by building one. Explain *why*,
  not just *what*. Ground every claim in `PLAN.md`; never invent APIs not in the plan.
- Use Docusaurus admonitions (`:::note`, `:::tip`, `:::warning`, `:::info`).
- Python/bash/json fenced code blocks; keep snippets consistent with `PLAN.md`.
- Cross-link related pages with relative links. Note that `src/*.py` are still stubs where
  relevant ("planned, not yet implemented") so docs don't overstate the code's state.
